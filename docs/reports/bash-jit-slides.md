## Slide 2: Bash 的性能困境

**标题：** Bash 的性能困境

**核心内容：**

Bash 是最广泛使用的脚本语言，但它有根本性的性能限制：

| 问题 | 说明 | 举例 |
|------|------|------|
| 解释执行 | 逐行解析，无编译优化 | 循环内变量操作比 Python 慢 100x |
| 进程创建开销 | 每次调用外部命令都 fork+exec | `for i in $(seq 5000); do tr ...; done` 耗时 6 秒 |
| 数据结构原始 | 数组和哈希表操作低效 | 字符串处理靠 `${var//pattern/replace}` |
| 字符串处理慢 | 每次操作重新解析 | `wc -l`、`awk`、`sed` 是唯一选择 |

**配图建议：** 左侧放一个简单的 Bash 循环代码片段（for 循环内调用 tr/head 生成密码），右侧放等价的 Python 代码（random.choices），中间用箭头和数字标注性能差距（6s vs 16ms）。

**演讲要点：** Bash 的"威力"来自调用外部命令，但这恰恰是它的性能瓶颈。循环中每次调用外部命令都要 fork+exec，这个开销在大量迭代时急剧放大。

## Slide 3: AI Agent 效率受限于 CLI

**标题：** AI Agent 的效率天花板：CLI 执行速度

**核心内容：**

```
用户任务 → AI Agent → 编写 Bash 命令/脚本 → 等待执行 → 解析结果 → 下一步
                                         ↑
                                    瓶颈在这里
```

- AI Agent（如 Claude Code、Cursor）大量操作通过 CLI 完成：文件搜索、代码分析、构建测试、日志处理
- 这些 CLI 操作本质上是 Bash 命令管道和脚本
- Agent 被迫等待慢速命令完成，浪费上下文窗口和计算资源
- Agent 无法为提升性能而重写脚本——必须使用宿主环境提供的方式

**配图建议：** 流程图展示 AI Agent 的工作循环（思考→编写命令→等待执行→读取结果→思考），用红色标注"等待执行"环节的时间消耗，引出问题：如果底层 Bash 执行能加速，Agent 整体效率就能提升。

**演讲要点：** AI Agent 的工作效率直接受限于它所使用的工具的执行速度。Bash 执行速度成为 AI Agent 效率的天花板。如果能透明加速 Bash，就能同时提升人类和 AI Agent 的工作效率。

## Slide 4: 核心洞察

**标题：** 核心洞察：让 Bash 自行加速

**核心内容：**

> 如果在 Bash 执行层透明地加速脚本，就能在不改变任何上层使用方式的前提下，同时提升人类和 AI Agent 的工作效率。

三个关键约束：

1. **零修改** — 现有脚本不需要任何改动
2. **零感知** — 上层（用户、Agent）完全不知道底层发生了什么
3. **零风险** — 任何失败静默回退到原始 Bash

**配图建议：** 中间放 Bash 层，上方是"用户 / AI Agent"（标注"无感知"），下方是 Bash JIT 编译层（标注"透明加速"），左右两侧分别是"未优化"和"已优化"的执行路径。强调上层调用方式完全不变。

**演讲要点：** 我们的思路不是让用户重写脚本，也不是让 Agent 改用 Python，而是在 Bash 内部透明地完成编译加速。用户或 Agent 执行 `bash script.sh` 的方式不需要任何改变。

## Slide 5: 方案概览

**标题：** Bash JIT：LLM 翻译 Bash → Python

**核心内容：**

利用大语言模型（LLM）将 Bash 脚本自动翻译为 Python，实现透明加速：

```
                    Bash 执行层
                         │
              ┌──────────┴──────────┐
              │                     │
         首次运行               第二次运行
              │                     │
     ┌────────┴────────┐            │
     │                 │            │
  Bash 原生执行    异步编译          │
  （正常速度）     Bash → Python     │
                     │              │
              写入本地缓存           │
              (~2-5秒完成)          │
                                    │
                         ┌──────────┴──────────┐
                         │                     │
                    缓存命中              execvp 替换
                    读取 compiled.py     python3 直接执行
```

- **编译工具**：LLM（Claude Sonnet），将 Bash 翻译为等价 Python
- **缓存策略**：FNV-128 指纹识别，本地磁盘缓存，同一脚本只编译一次
- **执行替换**：`execvp("python3", ...)` 进程级替换，无调度开销

**配图建议：** 按上面的流程图绘制，强调"首次运行"和"第二次运行"两条路径。

**演讲要点：** 首次运行时 Bash 正常执行，同时后台调用 LLM 编译。编译完成后写入缓存。第二次运行时，Bash 在解析脚本之前就把自己替换为 Python 进程，上层完全无感知。

## Slide 6: 系统架构

**标题：** 系统架构

**核心内容：**

```
┌──────────────────────────────────────────┐
│            Bash 进程 (C)                  │
│                                          │
│  bash_jit_try_script()                   │
│    1. 读取脚本，计算 FNV-128 指纹         │
│    2. 检查缓存（compiled.py 是否存在？）  │
│    3. 缓存命中 → execvp("python3", ...)  │
│    4. 缓存未命中 → 上报守护进程           │
│                                          │
└──────────────┬───────────────────────────┘
               │ Unix Domain Socket
               ▼
┌──────────────────────────────────────────┐
│       bash_jitd (Python 守护进程)         │
│                                          │
│  · 接收执行报告，维护全局计数器           │
│  · 整脚本请求：立即触发 LLM 编译          │
│  · 逐命令请求：计数达阈值后编译           │
│  · 翻译 Bash → Python，写入缓存          │
│  · 语法验证 + 安全检查                   │
│  · 最多 3 个并发编译                     │
└──────────────┬───────────────────────────┘
               │ HTTPS
               ▼
┌──────────────────────────────────────────┐
│           LLM API (Claude)               │
│  输入：Bash 源码                          │
│  输出：等价 Python 代码                   │
└──────────────────────────────────────────┘
```

**配图建议：** 按上面的架构图绘制，三个层次（Bash 进程、守护进程、LLM API）从上到下排列，用不同颜色区分。标注通信方式（Unix Socket、HTTPS）。

**演讲要点：** 架构分三层：Bash 进程内嵌的 C 代码负责拦截和缓存查找；独立的 Python 守护进程负责 LLM 调用和缓存管理；LLM API 负责实际的代码翻译。三层通过 Unix Socket 通信，Bash 进程不阻塞等待编译结果。

## Slide 7: 两种编译模式

**标题：** 整脚本 JIT vs 逐命令 JIT

**核心内容：**

| | 整脚本 JIT | 逐命令 JIT |
|---|---|---|
| **触发场景** | `bash script.sh` | 交互式 shell、`eval`、`source` |
| **拦截点** | `shell.c`，脚本解析之前 | `reader_loop()`，每条命令执行前 |
| **编译时机** | 首次执行立即编译 | 执行次数超阈值（默认 100 次）后编译 |
| **执行方式** | `execvp` 替换整个进程 | 构造 `python3` 命令替换原命令 |
| **缓存粒度** | 整个脚本文件 | 单条命令/函数 |

**配图建议：** 左右两栏对比布局，左侧"整脚本 JIT"，右侧"逐命令 JIT"，各配一个简短代码示例说明触发场景。

**演讲要点：** 整脚本是主要模式——对 AI Agent 和 CI/CD 场景最重要。逐命令模式适用于交互式场景，需要多次执行同一命令后才触发编译，避免对短命命令浪费 LLM 调用。

## Slide 8: 整脚本执行流程

**标题：** 整脚本 JIT：执行流程详解

**核心内容：**

**缓存命中路径（第二次运行）：**

```
bash script.sh
  │
  ├─ bash_jit_init(): BASH_JIT=1 → 初始化 JIT，连接守护进程
  │
  ├─ bash_jit_try_script("script.sh"):
  │    ├─ 读取文件 → 计算 FNV-128 指纹
  │    ├─ 检查 ~/.cache/bash_jit/<fp>/compiled.py → 存在！
  │    ├─ setenv("BASH_JIT_SCRIPT", "script.sh")
  │    └─ execvp("python3", ["python3", "compiled.py", "$@"])
  │         └─ 进程替换，bash 不再存在，Python 直接运行
  │
  └─ （不会到达这里）
```

**缓存未命中路径（首次运行）：**

```
bash script.sh
  │
  ├─ bash_jit_try_script("script.sh"):
  │    ├─ 检查 compiled.py → 不存在
  │    └─ 通过 Unix Socket 发送 {"type":"exec_script", "source":"..."} 给守护进程
  │         └─ 非阻塞发送，bash 继续正常执行
  │
  ├─ Bash 正常执行脚本（原始速度）
  │
  └─ 脚本结束，bash 退出
       │
       └─ 守护进程后台完成编译，写入 compiled.py 到缓存
          （下次运行即可命中缓存）
```

**配图建议：** 上下两个流程图，上面是"缓存命中"路径（简洁，突出 execvp），下面是"缓存未命中"路径（显示异步编译）。

**演讲要点：** 关键设计：`execvp` 直接替换进程，不是启动子进程。Python 进程继承 bash 的所有文件描述符、环境变量、工作目录，行为等价。首次运行时发送消息是非阻塞的，不影响 bash 正常执行。

## Slide 9: 性能对比 — 自有脚本

**标题：** 性能对比：自有脚本

**核心内容：**

**source-stats.sh — CPU 密集型（44.3x 加速）：**

| 版本 | 耗时 | 说明 |
|------|------|------|
| Bash 原生 | 16,639 ms | 解释执行，循环 + 字符串处理慢 |
| JIT → Python | 376 ms | Python 原生循环 + 字符串操作 |
| **加速比** | **44.3x** | |

- 递归扫描目录，分析代码行/注释行/空行/超长行，按扩展名聚合
- Python 翻译用 `os.scandir()` 替代 `find`，用 `collections.Counter` 替代 `sort | uniq`

**git-file-contributors.sh — I/O 密集型（1.15x）：**

| 版本 | 耗时 | 说明 |
|------|------|------|
| Bash 原生 | 49,211 ms | 对 1651 个文件逐一调用 `git log` |
| JIT → Python | 42,802 ms | 相同的 `git log` 子进程调用 |
| **加速比** | **1.15x** | 瓶颈在 git 子进程 I/O |

**配图建议：** 横向柱状图，两根柱子分别代表 Bash 和 Python 耗时，柱子上标注具体数值。CPU 密集型的柱子高度差距巨大（视觉冲击），I/O 密集型的柱子几乎等高。

**演讲要点：** JIT 对 CPU 密集型脚本效果最好——循环、字符串处理、数组操作等 bash 弱项恰好是 Python 强项。I/O 密集型脚本瓶颈在外部进程，Bash 和 Python 都要等待，加速有限。

## Slide 11: 加速效果分布

**标题：** 什么类型的脚本加速最多？

**核心内容：**

```
加速倍数
  │
  │  388x ── ● pass         （循环 + 子进程 → Python random）
  │  195x ── ● pickname     （循环 + 子进程 → Python random.sample）
  │   44x ── ● source-stats （CPU 密集 → Python 原生操作）
  │   1.8x ── ● anagrams    （循环 + 函数 + 管道）
  │   1.5x ── ● syllables   （循环 + 函数 + 管道）
  │   1.4x ── ● bigrams     （循环 + 文件 I/O）
  │   1.1x ── ● git-file    （I/O 密集，git log 瓶颈）
  │   1.0x ── ● count-words（已高效管道，无优化空间）
  │
  └──────────────────────────────────────────────
     循环+子进程    CPU密集     文件I/O     外部进程瓶颈
```

**关键规律：**

- **高加速（100x+）**：循环内调用外部命令 → Python 标准库替代，消除 fork+exec 开销
- **中等加速（1.5x–5x）**：CPU 密集操作 → Python 原生数据结构和算法
- **低加速（~1x）**：瓶颈在外部进程 I/O → Bash 和 Python 都要等待

**配图建议：** 散点图或横向条形图，X 轴是脚本类型（循环+子进程、CPU 密集、文件 I/O、外部进程瓶颈），Y 轴是加速倍数（对数刻度）。用颜色渐变表示加速程度。

**演讲要点：** 加速效果的核心决定因素是"Bash 的开销占比"。Bash 开销越大（进程创建、解释执行循环），JIT 加速越明显。如果脚本已经在调用高效的外部命令（git、curl），JIT 帮不上太多。

## Slide 12: 编译正确性

**标题：** 编译正确性：8 组脚本输出对比

**核心内容：**

| 测试用例 | Bash 输出 | Python (JIT) 输出 | 评价 |
|---------|----------|------------------|------|
| source-stats | 94 行 | 94 行 | 格式一致 |
| git-contributors | 5,871 行 | 5,871 行 | 完全一致 |
| rand-pass | 5000x32 字符 | 5000x32 字符 | 格式一致 |
| rand-pickname | 500 个文件 | 500 个文件 | 格式一致 |
| nlp-count-words | 2,940 行 | 2,940 行 | 完全一致 |
| nlp-bigrams | 9,646 行 | 9,645 行 | 行数差 1 |
| nlp-anagrams | 0 行 (缺 rev 命令) | 1,234 行 | JIT 优于基线 |
| nlp-syllables | 5 行 | 5 行 | 近似一致 |

**典型差异及原因：**

- **Tab vs Space 分隔符**：Bash 工具（`paste`）默认 Tab 分隔，Python 用空格。不影响数据内容。
- **排序稳定性**：相同键值的排序顺序可能不同（`sort -rn` vs Python `sorted`）。
- **音节计数**：近似算法，不同实现的启发式规则不同。
- **nlp-anagrams**：Bash 版因缺 `rev` 命令失败，Python 版正常工作——JIT 消除了对外部命令的依赖。

**结论：不存在语义错误，所有差异属于格式或环境差异。**

**配图建议：** 表格为主，最后一行用绿色高亮"无语义错误"的结论。nlp-anagrams 行可加标注"JIT 优势"。

**演讲要点：** 正确性是 JIT 的底线——如果翻译结果不对，加速毫无意义。8 组测试全部通过，差异均为格式、环境或算法近似导致。还有一个意外收获：Python 翻译消除了对外部命令的依赖，提高了脚本可移植性。

## Slide 14: 设计亮点

**标题：** 设计亮点

**核心内容：**

| 设计决策 | 实现 | 效果 |
|---------|------|------|
| 进程替换 | `execvp("python3", ...)` | 无调度开销，Python 直接继承 bash 的全部状态 |
| 首次执行即时编译 | 守护进程收到 `exec_script` 立即编译 | 无需等阈值，整脚本首次运行即触发 |
| 非阻塞上报 | `send(socket, ..., MSG_DONTWAIT)` | bash 不等待编译结果，正常执行不受影响 |
| 全局守护进程 | 每用户一个，Unix Socket 通信 | 多个 bash 进程共享编译结果和计数器 |
| FNV-128 指纹 | 仅基于文件内容计算 | 脚本修改后自动重新编译 |
| C 内嵌 + Python 守护进程 | 拦截在 C 侧，编译在 Python 侧 | 利用 bash 源码级集成 + Python 生态 |
| 语法验证 + 安全检查 | 编译后自动检查 Python 语法和危险操作 | 防止无效或危险代码写入缓存 |

**配图建议：** 表格布局，每行一个亮点，左侧是设计决策，中间是技术实现，右侧是带来的效果。

**演讲要点：** `execvp` 是整个系统的关键——不是启动子进程，而是直接替换当前进程。这意味着 Python 进程从 bash 那里继承了文件描述符、环境变量、工作目录、命令行参数，行为完全等价。

## Slide 16: 局限性与风险

**标题：** 诚实面对局限：开环编译的本质风险

**核心内容：**

**开环控制问题：编译过程缺乏"生成-验证"反馈环**

```
开环（Bash JIT 现状）：                   闭环（理想状态）：
                                          ┌──────────┐
Bash 源码 → LLM → Python → 直接使用      │ 输入 → 生成 → 验证 ── 通过 → 使用
                 ↑                        │         ↑      ↓
                 └── 无反馈 ←───┘         │         └── 不通过 ←┘
                                            └──────────┘
```

LLM 翻译是概率性的，不是确定性编译。编译结果的正确性完全依赖 LLM 单次生成质量，没有验证步骤确认 Python 输出与 Bash 行为一致。Python 脚本可能执行无异常、无错误返回，但行为与 Bash 原版不一致——这种"静默错误"无法通过常规手段检测。

| 脚本复杂度 | 翻译正确率 | 典型风险 |
|-----------|-----------|---------|
| 简单（循环+字符串） | 高 | 格式差异（Tab/Space、排序稳定性） |
| 中等（函数+管道） | 较高 | 边界条件差异（行数差 1、编码问题） |
| 复杂（trap/信号/fd） | 较低 | 行为语义偏差 |

**AI Agent 场景：开环编译脱离了 Agent 的闭环工作循环**

```
Agent 的闭环：生成 bash → 执行（被 JIT 替换为 Python）→ 观察结果 → 有错？→ 修改 → 重试
                                ↑                                              ↓
                                └─ JIT 开环编译在此插入，Agent 完全不感知 ───────┘
```

- Agent 具备闭环纠错能力（执行→观察→修改），但 JIT 的开环编译发生在这个闭环之外
- Agent 无法判断异常是自己的 bash 脚本问题，还是 JIT 编译引入的偏差
- "双层 LLM"误差叠加：脚本由 LLM 生成（第一层），编译由 LLM 完成（第二层），两层都缺乏验证

**核心矛盾：** 透明性将 JIT 的开环过程完全隐藏在 Agent 的闭环反馈之外，系统整体失去自我纠错能力。

**配图建议：** 左侧是开环 vs 闭环的对比图（简单流程图），右上角是正确率表格，右下角是 Agent 闭环被 JIT 开环打断的示意图。用红色标注"无反馈"和"脱离闭环"。

**演讲要点：** 这是 Bash JIT 最需要诚实面对的问题。从控制论角度看，LLM 编译是开环的——生成后没有验证步骤。在 AI Agent 场景中，这个开环过程还脱离了 Agent 自身的闭环工作循环，Agent 可能陷入"脚本正确但执行错误"的死循环。最根本的改进方向是为编译过程建立闭环验证。

## Slide 17: 从局限到新方向

**标题：** 从 Bash JIT 的局限推导新方向

**核心内容：**

Bash JIT 的开环问题不是实现缺陷，而是"Agent 写 CLI + 系统透明编译"范式的固有缺陷。CLI 的确定性假象掩盖了 LLM 编译引入的不确定性，Agent 无法感知和纠正。

**新范式：Agent 以意图级接口调用系统 Agent**

```
当前范式：
  Agent → 编写 Bash → 系统透明编译为 Python → 执行
                           ↑ 开环，脱离 Agent 反馈循环

新范式：
  Agent → 结构化意图请求 → 系统 Agent → 理解意图 → 最优执行流 → 返回结果
                                                      ↓
                              Agent 感知不确定性，纳入闭环反馈 ←┘
```

**为什么是操作系统？**
- OS 对系统效率理解最深——进程调度、内存管理、I/O 栈
- "统计仓库代码行数"→ OS 可以编译出并行化的最优执行流，而非机械执行 `find | xargs wc -l`

**配图建议：** 左右对比布局。左侧"当前范式"用红色标注开环断裂点；右侧"新范式"用绿色标注闭环反馈。

**演讲要点：** Bash JIT 的价值不仅在于 388 倍加速，更在于通过实践暴露了一个根本问题——CLI 作为 Agent 工具调用接口的固有缺陷。由此推导出的方向是：OS 提供意图级的系统 Agent 接口，内部编译最优执行流。

## Slide 18: 行业现状——都有 AI Shell，但没有人真正代替 CLI

**标题：** 行业现状：都有 AI Shell，但没有人真正代替 CLI

**核心内容：**

市面上的"Agent OS / AI Shell"方案可归为三类，**没有一类做到了"语义 → 最优执行流"**：

| 类型 | 代表方案 | 实际做的事 | 核心局限 |
|------|---------|-----------|---------|
| AI 包装 Shell | ANOLISA cosh、OSPilot、openEuler PolyMind | 自然语言 → 翻译成 bash 执行 | 底层仍走 spawn→bash→grep，优化的是"减少摸索"，不是执行效率 |
| MCP 工具注册表 | 统信 UOS-MCP | 底层能力封装为结构化工具集 | Agent 调的还是 tool call，tool 内部该 spawn 进程还是 spawn |
| 安全沙箱 | 微软 MXC、ANOLISA AgentSecCore | 身份、权限、隔离、审计 | 不让你跑得更快，只是让你摔不死 |

**为什么生态没有形成？核心在竞争力问题：**

| 因素 | 说明 |
|------|------|
| 增值不够厚 | 系统 Agent 的增值主要在安全/可观测性/减少摸索，不在执行效率质变 |
| 自然语言对 Agent 反而是退化 | Agent 已有结构化 tool call，不需要把精确意图拍扁成自然语言再让另一个 LLM 去理解 |
| 网络效应锁死在 CLI | 所有 Agent 框架都对接 bash/container exec，因为这是最小公分母 |

**结论："OS 提供意图级接口、内部编译最优执行流"——不是有人做砸了，是还没人真正动工。**

**演讲要点：** 现有的"AI OS"方案都停留在过渡形态——AI 包装 Shell + 安全沙箱 + 可观测性，回避了"OS 替 Agent 生成最优执行流"这个核心硬骨头。生态卡住的原因很冷酷：系统 Agent 在执行效率侧没有提供足够竞争力，上层 Agent 发现用它还不如自己直接调 CLI。Bash JIT 实测的两个数量级效率差距，恰恰指向了这个空白——谁能填补它，谁就定义了下一代 POSIX。

## Slide 19: 安全架构——最小权限原则

**标题：** 安全架构：系统 Agent 不能是特权 + 自然语言

**核心内容：**

**安全风险：** 如果系统 Agent 是特权进程且接受自然语言请求，LLM 的不确定性可能导致错误的特权操作，对系统造成严重破坏。

**解决方案：拆分为两层**

```
应用 Agent
  ↓ 自然语言
应用内系统 Agent（每应用一个）
  · 身份 = 应用 UID/GID/capabilities
  · 权限 = 应用权限（应用能干啥它就能干啥）
  · 受 LLM 不确定性影响，但无法突破应用权限边界
  ↓ 结构化 IPC（无自然语言）
全局系统 Agent 服务（可选）
  · 身份 = 系统服务
  · 不信任应用内 Agent，只暴露安全设计的 IPC 接口
  · 类似 syscall/D-Bus 的传统权限模型
```

**核心原则：**

- 应用内系统 Agent 是为了帮助应用 Agent 实现更丰富的功能和更高的效率，不是用来提供特权能力
- 它与当前 Bash 脚本的安全模型一致：脚本可能有 bug，但权限不会超过调用者
- 全局系统 Agent 服务是否需要存在，取决于原有 syscall 和系统服务接口是否已足够

**配图建议：** 上方是安全风险示意（红色：特权进程 + 自然语言 = 危险），下方是两层架构图（绿色：应用内 Agent 权限受限，系统 Agent 服务无自然语言接口）。

**演讲要点：** 自然语言 + 特权是一个危险组合。解决方案是让 LLM 驱动的 Agent 运行在应用的权限边界内——无论 LLM 生成什么指令，都无法突破应用的权限。这和 Bash 脚本的安全模型完全一致：脚本可能有 bug，但它的权限不会超过调用者。全局系统 Agent 服务只暴露结构化 IPC 接口，不存在"LLM 误解自然语言导致特权操作"的风险。

## Slide 21: 总结与展望

**标题：** 总结

**核心内容：**

**Bash JIT 的价值：**

- **透明加速**：上层无感知，执行方式不变，`bash script.sh` 的用法完全一样
- **数量级提升**：对 CPU 密集型和循环+子进程型脚本，加速 44x–388x
- **安全回退**：任何失败静默回退，JIT 永远不会让脚本更慢或更不正确
- **AI Agent 友好**：底层加速 Bash，上层 Agent 效率直接提升

**需要诚实面对的局限：**

- LLM 编译是开环过程——缺乏"生成-验证"反馈环，正确性无法保证
- 透明性将开环编译隐藏在上层闭环之外，AI Agent 可能陷入调试死循环
- 编译结果不可复现，同一脚本不同时间编译可能产生不同行为

**关键数据：**

| 指标 | 数值 |
|------|------|
| 最高加速 | 388x（循环+子进程） |
| 平均加速（CPU 密集） | 44x |
| 编译正确性 | 8/8 脚本通过，无语义错误 |
| 代码修改量 | 0（现有脚本不需要任何改动） |

**未来方向：**

- **意图级 Agent-OS ABI**：OS 提供结构化的意图级接口（超越纯自然语言），内部编译最优执行流，取代 CLI 成为 Agent 工具调用的事实标准——这是从 Bash JIT 的局限推导出的根本方向，也是当前行业的空白
- 双重执行采样验证：首次编译后同时运行 Bash 和 Python 版本对比输出（但一次通过不代表永远正确，且非幂等脚本不能执行两次）
- 适用范围限制：仅对高置信度的简单脚本自动启用 JIT
- Agent 感知机制：让 Agent 感知 JIT 的存在，将 JIT 纳入 Agent 的闭环反馈中
- 确定性编译：固定 LLM 随机种子，使同一脚本始终产生相同 Python 代码

**配图建议：** 上半部分是三个核心价值点（透明、加速、安全），下半部分是关键数据表格。

**演讲要点：** Bash JIT 证明了 LLM 可以作为编译器的后端，将一种语言翻译为另一种更高效的语言。但 LLM 编译是开环的——没有验证步骤，正确性无法保证。而由于程序语义等价是不可判定问题，即使加入双重执行采样验证，也只是概率性的缓解，不是根本解决。透明性将这个开环过程隐藏在上层闭环之外，系统整体失去自我纠错能力。从这个局限出发，结合对行业现状的分析——现有的"AI OS"方案都停留在过渡形态，没有一家做到了"语义 → 最优执行流"，生态因竞争力不足而未形成——我们推导出一个更根本的方向：OS 应该提供意图级的系统 Agent 接口（结构化的 Agent-OS ABI），取代 CLI 成为 Agent 工具调用的事实标准。谁先做出来一个可审计、可回滚、兼容现有生态、又确实把执行效率提升一个数量级的方案，谁就定义了下一代 POSIX。
